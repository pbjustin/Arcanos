# GPT-OSS Durable Replay Store Implementation Plan

Phase 5.6 defines the implementation plan for the future GPT-OSS private
serving durable replay store. This phase does not implement the live store. It
adds a design-only migration draft, a TypeScript-owned interface contract shape
in JavaScript, validation gates, and rollback criteria.

This plan does not connect to a live database, apply migrations, create a
server, expose endpoints, call OpenAI, train, use vLLM, run Railway commands,
or use real secrets.

## Purpose

The durable replay store will eventually persist accepted signed request
nonces so replay protection survives process restarts and multiple private
serving workers. Phase 5.6 prepares that implementation without enabling it.

Current required readiness remains:

```json
{
  "replayProtectionDurableDesigned": true,
  "replayProtectionDurableImplemented": false,
  "replayProtectionDurable": false,
  "durableReplayMigrationDraftReady": true,
  "durableReplayMigrationApplyAllowed": false,
  "durableReplayMigrationApplied": false,
  "privateServingImplemented": false,
  "privateServingExposed": false,
  "cloudReady": false,
  "customGptReady": false
}
```

## Planned Storage Boundary

The future storage table is drafted at:

```text
migrations/drafts/gptoss_durable_replay_store.sql
```

The draft is explicitly marked design-only and must not be applied. A later
phase must convert it into a reviewed migration only after the serving boundary,
auth integration, operational database target, and rollback controls are
approved.

Phase 5.7 adds a migration guard for review/preflight only. The migration draft
exists only as a blocked draft under `migrations/drafts/`. No reviewed
executable migration has been promoted, no migration apply path exists, and no
durable replay migration execution is available.

Planned table shape:

| Column | Purpose |
| --- | --- |
| `id` | Row identifier generated by the database or migration layer. |
| `key_id` | Signing key id used for replay scope. |
| `nonce_hash` | SHA-256 hash of the nonce; raw nonce is never stored. |
| `request_id` | Signed request id for audit correlation. |
| `body_hash` | SHA-256 hash of the signed request body. |
| `first_seen_at` | First accepted request timestamp for this nonce hash. |
| `expires_at` | Expiration time for pruning. |
| `replay_window_seconds` | Window used to compute expiration. |
| `audience` | Expected signed request audience. |
| `subject` | Authenticated subject when available. |
| `source` | Source marker such as `gptoss_private_serving`. |
| `created_at` | Record creation timestamp. |

Required uniqueness:

```text
unique(key_id, nonce_hash)
```

The future store must reject conflicts on that pair before any model/runtime
behavior is invoked.

## Interface Contract

The contract module is:

```text
scripts/gptoss/private-serving/private-serving-durable-replay-store.mjs
```

It exports only:

- `createDurableReplayStoreContract()`
- `validateDurableReplayRecordShape(record)`
- `normalizeReplayStoreRecord(input)`
- `buildReplayStoreInsertPlan(decision)`

The module may normalize input and hash a nonce into `nonce_hash`, but it must
not return raw nonce values in normalized records or insert plans. It must not
import database clients or execute SQL.

## Validation Gate

The Phase 5.6 gate is:

```bash
npm run gptoss:private-serving:durable-replay:implementation-plan:validate
```

The Phase 5.7 migration guard is:

```bash
npm run gptoss:private-serving:durable-replay:migration-guard
```

The guard validates the draft only. It does not apply SQL, dry-run a migration,
execute a migration runner, open database connectivity, or enable durable
replay.

The validator checks:

- implementation plan exists
- design-only migration draft exists
- draft contains `DO NOT APPLY`
- draft contains `NO LIVE DB EXECUTION`
- draft uses `nonce_hash` and does not define a raw nonce column
- draft declares `unique(key_id, nonce_hash)`
- contract module has no database client imports
- no live database URL environment variable usage
- no Railway command usage
- no server/listener pattern
- readiness remains blocked for serving and exposure
- migration apply remains blocked with `applyAllowed:false`
- live DB writes remain disabled with `liveDbWrite:false`

Reports are written only under:

```text
local_artifacts/gptoss-runtime/
```

## Failure Modes

The future implementation must fail closed on:

- missing durable replay store
- write timeout or ambiguous write result
- unique conflict for `key_id + nonce_hash`
- stale timestamp or future timestamp
- invalid nonce shape
- missing `key_id`
- migration/schema version mismatch
- pruning failure that threatens capacity

Phase 5.6 only documents and validates those requirements. It does not provide
runtime fallback behavior for exposed serving.

## Rollback Plan

If a future durable replay implementation fails validation, disable GPT-OSS
private serving rather than downgrading to non-durable replay checks.

Rollback must preserve:

- no public endpoint exposure
- no Custom GPT exposure
- no raw request body storage
- no raw nonce storage
- no secret storage
- no OpenAI, training, vLLM, Railway, live database, shell, or server fallback

Post-rollback gates:

```bash
npm run gptoss:private-serving:durable-replay:implementation-plan:validate
npm run gptoss:runtime:cloud-gate
npm run gptoss:runtime:release-gate:ci
```

The expected post-rollback readiness state remains `cloudReady:false` and
`customGptReady:false`.
